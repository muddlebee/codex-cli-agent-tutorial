# m3 Tutorial: Migrate to Codex App-Server JSON-RPC

## Learning Objective

By the end of this milestone, you can run the same CLI UX on top of persistent Codex thread/turn RPC APIs.

## Starting Point

```bash
git checkout m3
npm install
npm run lint
npm test
```

## Why m3 Exists

`codex exec` is great for quick wins but limited for long-lived orchestration.
RPC gives you:
- explicit thread lifecycle
- turn-level operations
- richer event streams

## What Gets Added

- `JsonRpcClient` over stdio
- `CodexRpcProvider` implementing session operations
- provider factory:
  - `NANO_PROVIDER=exec` (default)
  - `NANO_PROVIDER=rpc`

## Important Files

- `src/providers/codex-rpc/jsonrpc-client.ts`
- `src/providers/codex-rpc/provider.ts`
- `src/providers/factory.ts`
- `src/cli/chat.ts` (session-aware provider usage)

## Step-by-Step Walkthrough

1. Bootstrap RPC client
- spawn `codex-app-server`
- send `initialize`
- send `initialized`

2. Session lifecycle
- `thread/start` for new thread
- `thread/resume` for existing thread

3. Turn execution
- call `turn/start`
- drain notifications
- map notifications to `RuntimeEvent`

4. Keep CLI stable
- chat/run interface stays the same
- only provider internals switch

## Run It

```bash
NANO_PROVIDER=rpc npm run dev -- chat
```

## Expected Output Pattern

- same style as exec mode, but backed by RPC notifications
- persistent remote thread id associated to local session meta

## Troubleshooting

1. `codex-app-server` not found
- set `CODEX_APP_SERVER_BIN=/path/to/codex-app-server`

2. `thread/start did not return a thread id`
- check app-server version compatibility

3. no notifications arrive
- verify initialize handshake completed

## Exercises

1. Easy: expose current provider mode in prompt header.
2. Medium: add `thread/list` command.
3. Hard: validate JSON-RPC payloads with schemas.

## Recap

You now have persistent Codex-native control flow.
In m4, you add approval responses and in-flight turn control.
