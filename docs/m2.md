# m2 Tutorial: Planning Layer with Todo Constraints

## Learning Objective

By the end of this milestone, you can guide agent behavior with explicit todo state and enforce planning constraints.

## Starting Point

```bash
git checkout m2
npm install
npm run lint
npm test
```

## Why m2 Exists

Conversation memory alone is not planning.
This milestone introduces a controlled planning surface so long tasks stay structured.

## What Gets Added

- session-scoped `TodoManager`
- todo file persisted per session
- slash commands in chat:
  - `/todo add <text>`
  - `/todo start <id>`
  - `/todo done <id>`
  - `/todo list`
- todo context injected into prompts

## Important Files

- `src/core/todo-manager.ts`
- `src/cli/todo.ts`
- `src/cli/chat.ts`

## Constraints (Intentional)

- max 20 todo items
- only one `in_progress` at a time

These constraints keep plans simple and force focus.

## Step-by-Step Walkthrough

1. Todo persistence
- todos stored at `.nano-agent/sessions/<id>/todo.json`

2. Command handler
- parse `/todo` actions
- call manager methods
- return user-friendly status lines

3. Prompt injection
- render current todo list
- append to prompt before sending turn

## Run It

```bash
npm run dev -- chat
# in chat:
/todo add build rpc provider
/todo add test approval flow
/todo start 1
/todo list
```

## Expected Output Pattern

- todo IDs increment (`#1`, `#2`, ...)
- starting a second active task fails with constraint error
- `todo.json` reflects command actions

## Troubleshooting

1. `only one in_progress`
- complete current active todo before starting next

2. `todo limit reached`
- close or clean existing items

3. slash command ignored
- ensure exact format `/todo ...`

## Exercises

1. Easy: show pending count in chat prompt line.
2. Medium: add `/todo reopen <id>`.
3. Hard: add optional dependencies and detect cycles.
